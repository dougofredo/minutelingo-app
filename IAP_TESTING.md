# Testing In-App Purchases Locally

## Short answer: simulators are limited

- **iOS Simulator:** Real in-app purchases (sandbox) **do not work** with `react-native-iap`. StoreKit has no real App Store on the simulator, so you get errors like "Cannot connect to iTunes Store" or StoreKit errors. You **cannot** fully test the purchase flow on the iOS Simulator.
- **Android Emulator:** Can work **only if** you use an emulator image **with Google Play** (not just "Google APIs"), your app is on an **internal test track**, and the emulator is signed in with a **license tester** account. Many developers find a **physical Android device** more reliable.

So for real end-to-end IAP testing, use **physical devices** (or, for Android, a Play-enabled emulator + test track).

---

## What you can do

### 1. Run the app locally (simulator/emulator)

You need a **development build** (Expo Dev Client), not Expo Go, because IAP uses native code:

```bash
# iOS Simulator (IAP will not complete; you can test UI and that the app doesn’t crash)
npx expo run:ios

# Android Emulator (IAP may work if you use a “Play Store” image and internal test track)
npx expo run:android
```

On **iOS Simulator**, the IAP flow will typically fail when it tries to connect to the store (e.g. no products, or init error). That’s expected. You can still:

- Navigate to the subscription screen and see the UI.
- Confirm the app handles errors without crashing (your code already skips or handles missing products).

### 2. Test real purchases: use a physical device

**iOS (recommended):**

1. Create a **Sandbox** tester in [App Store Connect](https://appstoreconnect.apple.com/) → **Users and Access** → **Sandbox** → **Testers**.
2. On the **iPhone**: **Settings → App Store** → sign out of your normal Apple ID (do **not** sign in with the sandbox account in Settings).
3. Build and install the app on the device (e.g. `npx expo run:ios --device` or install via TestFlight).
4. In the app, go to **Account** → **Purchase Premium**. When the purchase sheet appears, sign in with the **sandbox** Apple ID. Complete the purchase (no real charge).
5. Confirm premium unlocks and, if you use a backend, that `profiles.has_access` is set.

**Android:**

1. In [Google Play Console](https://play.google.com/console/), add test Gmail accounts as **License testers** (Settings → License testing).
2. Upload the app to at least the **Internal testing** track and install it from the Play Store (or a Play-enabled emulator signed in with a license tester).
3. In the app, purchase premium. No real charge for license testers.
4. Confirm premium unlocks and backend update.

### 3. Optional: StoreKit Configuration (Xcode only, no backend)

For **UI flow only** on the **iOS Simulator** (no real receipts, no backend):

1. Open the iOS project in Xcode (e.g. `ios/` after `npx expo prebuild`, or open the workspace generated by `expo run:ios`).
2. **File → New → File** → **StoreKit Configuration File**. Add a product with ID `com.minutelingo.app.premium` (non-consumable).
3. In the **Run** scheme: **Edit Scheme** → **Run** → **Options** → set **StoreKit Configuration** to that file.
4. Run the app from Xcode on the simulator. StoreKit will use the local config so the simulator can “complete” a purchase, but the receipt is only valid in that test environment — your backend (Apple verifyReceipt) will not accept it as a real sandbox receipt. Use this for UI/flow only.

---

## Summary

| Environment              | Real IAP (sandbox/Play)? | What you can test                          |
|--------------------------|---------------------------|--------------------------------------------|
| iOS Simulator            | No                        | UI, error handling, no crash on init fail  |
| iOS physical + Sandbox   | Yes                       | Full purchase + restore + backend          |
| Android Emulator (Play)  | Maybe                     | Full flow if on internal track + tester    |
| Android physical         | Yes                       | Full purchase + restore + backend          |

So: **you can’t reliably test that in-app purchases “work” end-to-end on the simulator.** Use a **real device** (and, for Android, optionally a Play-enabled emulator with internal test track) for that.
